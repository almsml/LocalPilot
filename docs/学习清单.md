## 三、跨平台踩坑知识（必读）

### mDNS 在 Windows 上的问题

| 平台 | 情况 |
|------|------|
| **macOS** | Bonjour 原生支持，最稳定 |
| **Linux** | Avahi 守护进程，一般没问题 |
| **Windows** | IPv6 多播 `setsockopt` 报错、防火墙默认拦截 UDP 5353、多网卡时可能发错接口 |

**相关链接**：

- 🔗 [hashicorp/mdns Issue #80 — Windows 无结果](https://github.com/hashicorp/mdns/issues/80) — 错误日志：`listen udp6 [ff02::fb]:5353: setsockopt: not supported by windows`
- 🔗 [hashicorp/mdns Windows 兼容性问题汇总](https://github.com/hashicorp/mdns/issues?q=is%3Aissue+windows) — 多网卡、防火墙等常见问题
- 🔗 [IETF: 多播在各平台的支持现状](https://datatracker.ietf.org/meeting/121/materials/slides-121-mboned-survey-on-the-state-of-multicast-support-00) — Windows 多播支持最弱

**Phase 0 验证任务**：

1. 在 Windows 设备上跑 `hashicorp/mdns` 的 Server + Lookup 示例，看能否正常发现
3. 如果失败：把网络设为「专用网络」→ 允许防火墙 UDP 5353 → 明确指定网络接口
4. 如果还不行：规划里准备手动 IP 输入备选方案

### gRPC vs HTTP 取舍

搞懂这几个问题（面试大概率会问）：

- **gRPC 比 HTTP+JSON 好在哪？** → 强类型契约（编译期检查）、原生 streaming、二进制序列化体积小 3-10 倍
- **gRPC 比 HTTP+JSON 差在哪？** → 调试不方便（不能直接在浏览器里看）、需要 proto 编译步骤、浏览器不能直接调 gRPC 服务
- **什么时候不该用 gRPC？** → 需要浏览器直接调用的接口（所以 Dashboard ↔ Controller 走 HTTP/WS）

🔗 [gRPC 官方对比文档](https://grpc.io/docs/what-is-grpc/introduction/)

### Go ↔ Go gRPC 互通验证

Phase 0 已完成的验证：

1. Agent 和 Controller 用同一份 `.proto` 文件
2. 两边用 `protoc-gen-go` + `protoc-gen-go-grpc` 生成代码
3. 已跑通：Register + Heartbeat unary 调用

🔗 [Protocol Buffers 官方文档](https://protobuf.dev/)<br>

### 分布式系统基础概念

| 概念 | 在 LocalPilot 中的体现 |
|------|----------------------|
| **服务发现** | Agent 启动 → mDNS 广播 → Controller 发现 |
| **心跳** | Agent 每 5s gRPC Heartbeat → Controller 判断存活 |
| **调度** | Controller 多维打分 → 选最优设备执行任务 |
| **故障迁移** | 设备 30s 无心跳 → 任务重新入队 → 分配到其他设备 |
| **断点恢复** | Agent 定期上传 checkpoint → 新 Agent 从中断处继续（Phase 4） |

🔗 [Designing Data-Intensive Applications (Martin Kleppmann)](https://www.oreilly.com/library/view/designing-data-intensive-applications/9781491903063/) — 分布式系统必读，面试和实战都用的上<br>
🔗 [分布式系统 MIT 6.824 课程](https://pdos.csail.mit.edu/6.824/) — 如果想深入原理

### ffmpeg 基础（用于 Demo 场景）

- 基本命令：`ffmpeg -i input.mp4 -c:v libx265 -vf scale=1920:1080 output.mp4`
- 关键帧概念：为什么 `-c copy` 切割只能在 I 帧上切
- `segment` muxer vs 手动 `-ss` + concat 的区别
- 为什么切分合并后音画可能不同步，以及怎么应对

🔗 [ffmpeg 官方文档](https://ffmpeg.org/documentation.html)<br>
🔗 [ffmpeg Seeking 指南](https://trac.ffmpeg.org/wiki/Seeking) — 讲 `-ss` 和 `-c copy` 的微妙关系<br>
🔗 [ffmpeg Concatenate 指南](https://trac.ffmpeg.org/wiki/Concatenate) — concat demuxer vs concat filter 的区别<br>
🔗 [ffmpeg Segment Muxer 文档](https://ffmpeg.org/ffmpeg-formats.html#segment_002c-stream_005fsegment_002c-ssegment) — 用 `-f segment` 替代手动切分<br>
🔗 [SO: ffmpeg 切分合并音画同步问题 + 解决方案](https://stackoverflow.com/questions/79264444/) — 2024 年 12 月的完整讨论
