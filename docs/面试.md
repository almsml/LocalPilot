# LocalPilot 面试展示文案

> 面试官通常给你 3-5 分钟介绍项目，然后追问 10-15 分钟。这份文案按时间线组织——开场 30 秒抓住注意力，然后逐层深入。

---

## 一、开场（30 秒——必须背熟）

> 「我做了一个叫 LocalPilot 的个人设备网格计算平台。一句话就是——**把你吃灰的旧电脑、旧手机变成你的私人计算集群**。」
>
> 「比如我有一台旧 MacBook 和一台旧笔记本，平时闲置着。我写了一个系统，让它们能自动组网——一台设备做 Controller 负责调度，其他设备装 Agent 负责执行。我从浏览器里提交一个视频转码任务，系统会自动选负载最低的设备去跑，实时日志流回浏览器。如果跑着跑着那台设备挂了——30 秒内任务会自动迁移到另一台设备继续跑。」
>
> 「技术栈是 Go + gRPC + React。整个项目 5 个 Phase 从原型到容错全部做完，代码在 (GitHub 链接)。」

**要点**：不要念 PPT，用"我有一个问题 → 我做了个东西解决它"的叙事结构。数字（30 秒迁移、5 个 Phase）让面试官有追问的锚点。

## 二、架构图（边说边画——1 分钟）

在白板上画这个（提前练 3 遍）：

```
┌──────────────────────────────────────────┐
│           Dashboard (React)              │
│      浏览器 · HTTP/2 + WebSocket          │
└──────────────────┬───────────────────────┘
                   │
┌──────────────────▼───────────────────────┐
│           Controller (Go · 主力机)        │
│  设备注册 · 心跳检测 · 任务调度 · REST API │
└──────────────────┬───────────────────────┘
                   │ gRPC
      ┌────────────┼────────────┐
┌─────▼─────┐ ┌────▼─────┐ ┌───▼───────┐
│ Agent(Go) │ │Agent(Go) │ │ Agent(Go) │
│ 旧 MacBook │ │ 旧笔记本   │ │  旧手机    │
└───────────┘ └──────────┘ └───────────┘
```

边画边讲：

> 「三个组件。**Dashboard** 是用户看到的网页。**Controller** 是大脑——它维护设备注册表、做任务调度、检测心跳超时。**Agent** 是手脚——每台计算设备上跑一个，上报自己的 CPU/内存状态、接收 Controller 下发的任务、在沙箱里执行。」

> 「通信分两层：**Dashboard ↔ Controller 走 HTTP+JSON**，因为浏览器天然支持，调试方便。**Controller ↔ Agent 走 gRPC+Protobuf**，因为后端服务间通信需要强类型契约和流式传输——Agent 执行命令的时候，stdout 是通过 gRPC server-streaming 实时推回 Controller 的。」

## 三、技术亮点（挑 3 个讲——3 分钟）

根据面试官的背景，选 3 个最相关的：

### 亮点 1：多维调度器（面后端/基础设施岗位必讲）

> 「调度器不是随机选设备的。我对每台在线设备做**五维度综合打分**——」
>
> 在白板上写：
> ```
> Score = CPU空闲×0.35 + 内存空闲×0.25 + 架构匹配×0.20
>       + 延迟×0.10 + GPU×0.10
> ```
>
> 「CPU 和内存是通用计算核心资源，占 60%。架构匹配保证 ARM 任务不会发到 x86 设备。延迟用心跳时间差估算——心跳在 5 秒内的得满分，超过 15 秒的得低分。GPU 是稀缺资源——有 GPU 的设备得分更高，让无 GPU 设备留给不需要 GPU 的任务。」
>
> 「为什么不是简单的'选负载最低的'？因为单一维度会出问题——比如一台空载但网络不稳定的设备，得分可能低于一台轻度负载但连接稳定的设备。多维加权让每个维度之间做合理的权衡。」

**面试官可能的追问**：权重怎么来的？→ "初始值来自经验，代码里预留了配置化接口，可以从实际运行数据中通过梯度下降拟合最优权重。但说实话——对于个人设备的规模，硬编码的权重已经够用了。"

### 亮点 2：任务迁移（面基础设施/分布式岗位必讲）

> 「设备离线后 30 秒，Controller 会自动把任务迁移到其他设备。这个流程是这样的——」
>
> 在白板上画时间线：
> ```
> Agent 心跳断了
>   │
>   ├─ 15s ──→ UNHEALTHY（黄色警告，给设备自救机会）
>   │
>   └─ 30s ──→ OFFLINE（红色确认，触发迁移）
>                │
>                ├─ 回调通知 Job Manager
>                ├─ 查找该设备上所有 RUNNING/ASSIGNED 任务
>                ├─ 重置为 QUEUED，重新入队
>                ├─ 调度器重新选设备
>                └─ gRPC 下发到新设备执行
> ```
>
> 「为什么分两步——UNHEALTHY 和 OFFLINE？因为 WiFi 波动很常见。如果一心跳断了就迁移，可能刚迁移完原设备又恢复了——任务在两边同时跑。UNHEALTHY 给了 15 秒的'冷静期'。」
>
> 「迁移后任务从零开始执行，不做 checkpoint 恢复。这是故意做的 tradeoff——checkpoint 的工程复杂度太高了，对于几分钟到几十分钟的个人计算任务，从头执行比重启 checkpoint 更简单也更可靠。」

### 亮点 3：跨平台降级设计（面全栈/系统岗位必讲）

> 「这个系统要跑在 Windows、macOS、Linux 上——旧 MacBook 是 Intel 的，旧笔记本可能是 ARM 的。所以大量地方做了**分层降级设计**。」
>
> 「举两个例子——」
>
> 「**沙箱**：优先用 Docker → Docker 不可用就进程级 os/exec → 连 exec 都失败就返回错误。每层降级自动检测，用户不用选择。当前 Phase 2 实现了进程级执行——虽然安全性不如 Docker，但在'自己用自己的设备'这个场景下足够了。」
>
> 「**系统监控**：优先用 gopsutil 跨平台库 → Linux 上 gopsutil 失败就解析 /proc/meminfo → macOS 上就用 sysctl → Windows 上就用 kernel32.dll 的 GlobalMemoryStatusEx。每层都有 fallback，保证在任何平台上至少能拿到基础数据。」

## 四、Demo 演示脚本（2 分钟——如果有机会现场演示）

### Demo 1：基础任务执行（30 秒）

```
1. 打开 Dashboard http://localhost:5173
2. 切到"任务管理" Tab
3. 输入: echo "hello from LocalPilot"
4. 点执行 → 终端实时显示输出 → 状态变为 COMPLETED
```

边操作边说：
> 「刚才这个 echo 命令经过了完整链路：Dashboard → Controller REST API → 入队 → 调度器选设备 → gRPC Execute → Agent 沙箱执行 → gRPC stream 日志 → Dashboard 终端显示。全程约 1 秒。」

### Demo 2：故障迁移（1 分钟）

```
1. 提交一个长时间任务: ping -n 20 127.0.0.1
2. 任务开始执行，终端实时显示 ping 输出
3. 切到"设备列表"，看到设备状态 ONLINE（绿色）
4. 在终端里 Ctrl+C 关掉 Agent
5. 15 秒后设备变 UNHEALTHY（黄色）
6. 30 秒后设备变 OFFLINE（红色）
7. 任务自动在其他设备上重新开始执行
```

边操作边说：
> 「注意——我没有做任何手动操作。Dashboard 上设备状态自动流转，任务自动迁移。整个故障恢复过程对用户透明。」

## 五、常见追问 + 参考答案

### Q: 这个项目和 Ansible/SaltStack 有什么区别？

> 「Ansible 是配置管理——确保 100 台服务器的 nginx 配置一致。LocalPilot 是**计算调度**——把视频转码任务分配到最合适的设备上执行。一个关注状态收敛，一个关注任务分配。定位完全不同。」

### Q: 为什么不直接用 Kubernetes？

> 「K8s 假设你的节点是同构的云端 VM——64 核 128G 内存，有 Docker，有稳定的数据中心网络。LocalPilot 面对的是**极度异构的个人设备**——一台有 GPU，一台是 ARM，一台 Windows 不支持 Docker。而且 K8s 的最小部署复杂度远超个人场景——etcd、API server、controller manager——一个人在家搭一套 K8s 来跑 ffmpeg 转码？太重了。」

### Q: 系统能承受多少设备？

> 「mDNS 协议在同一局域网建议不超过 100 台。SQLite WAL 模式在数百台设备的心跳写入下没问题。实际测试目标是 10 台设备稳定运行。这不是云服务级别的系统——是**个人/小团队的网格计算工具**。」

### Q: 如果 Controller 挂了怎么办？

> 「Controller 是单点故障——这是有意为之。Controller 跑在主力机上——用户正在用的设备。如果主力机挂了，用户自己能感知到。而且 LocalPilot 的场景是用户主动提交任务——Controller 挂了就没有新任务，旧任务在 Agent 上跑完也没地方回报结果。所以在当前场景下，Controller 的单点不是问题。」

### Q: 你怎么保证任务隔离的安全性？

> 「多层降级：Docker 可用 → 容器隔离（只读根文件系统 + 网络隔离）。Docker 不可用 → 进程级 os/exec + setrlimit 资源限制。但需要坦诚地说——当前的进程级隔离不是安全产品级别的，`rm -rf /` 防不住。LocalPilot 的威胁模型是'自己用自己的设备'，不是公共平台跑不可信代码。如果威胁模型变了，架构预留了 DockerSandbox 扩展点。」

### Q: 这个项目你最大的收获是什么？

> 「最大的收获不是学会了 gRPC 或 React——而是学会了**在工程中做 tradeoff**。举三个例子——」
>
> 「**Agent 为什么从 Rust 换到 Go？** 最初规划是 Rust——性能好、内存安全。但实际开发中，Rust 的学习曲线严重拖慢进度——所有权、生命周期、Pin/Unpin。决定换到 Go 后，一周写完了 Rust 一个月没写完的代码。这个决策的核心是——**用务实换进度**。」
>
> 「**为什么不做 checkpoint？** 最初规划里有断点恢复功能。但深入分析后发现——checkpoint 需要 Agent 理解任务的中间状态、需要网络传输几百 MB 的快照、需要校验和防损坏。对于几分钟的任务来说，从头执行比重启 checkpoint 更简单。这个决策的核心是——**不盲目追求分布式系统的'完整度'，而是选择对场景最合适的方案**。」
>
> 「**为什么用 SQLite 而不是 Postgres？** 刚开始也想用 Postgres——更"专业"。但一想——设备注册表的数据量才几百行，部署 Postgres 要多维护一个服务、多一个故障点。SQLite 零运维，文件拖走就是备份，WAL 模式支持几十台设备的并发读写绰绰有余。这个决策的核心是——**选最简方案，不为'看起来更专业'而增加复杂度**。」

## 六、结尾话术

> 「这个项目如果让我继续做，有三个方向——」
>
> 「**NAT 穿透**：用 Tailscale 做 overlay network，让不在同一 WiFi 的设备也能组队。」
>
> 「**WASM 微隔离**：用 WebAssembly 替代 Docker 做沙箱，毫秒级冷启动，比容器轻 100 倍。」
>
> 「**移动端 Worker**：让旧 Android 手机通过 Termux 也变成计算节点。」
>
> 「如果团队有相关方向的需求，我可以深入讲任何一个模块的实现细节。」
