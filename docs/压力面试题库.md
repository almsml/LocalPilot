# LocalPilot 压力面试题库

> 每个问题都是面试官可能追问的第二、第三层——不是"你用了什么技术"，而是"为什么这样设计？如果换一种做法会怎样？你的方案有什么缺陷？"

---

## 一、mDNS 与 gRPC 通信

### Q1: mDNS 只能在局域网内工作。如果你的设备分布在不同 WiFi 下（比如一台在家、一台在公司），mDNS 就失效了。你怎么解决？

**面试官在考什么**：你不只理解了 mDNS 的优点（零配置），也理解它的局限（单播域）。以及你是否想过扩展方案。

**回答**：

> mDNS 确实受限于同一个广播域。LocalPilot 定位是"个人设备网格"，所以假设所有设备在同一局域网内是合理的——家里所有设备通常连同一个路由器。
>
> 如果要做跨网络，有三个方案按复杂度递增：
>
> 1. **手动 IP 输入**（Phase 1 备选方案）：Agent 启动时如果 mDNS 不可用，用户手动指定 Controller IP。最简单的 fallback。
> 2. **Tailscale/WireGuard overlay network**：所有设备加入同一个 Tailscale 网络，mDNS 在 overlay 上仍然工作，且自带 NAT 穿透和加密。这是我 Phase 4 计划中的 NAT 穿透方案。
> 3. **STUN/TURN + 信令服务器**：类似 WebRTC 的做法——需要一个公网信令服务器交换 IP:Port，然后 P2P 打洞。复杂度高，但不需要 VPN。
>
> 目前 Phase 1 实现了方案 1 的 fallback，Phase 4 预留了方案 2 的扩展点。

### Q2: 你说 gRPC 比 HTTP+JSON 好，但 gRPC 调试不方便——你不能在浏览器里直接看 gRPC 请求。如果线上出了问题，Agent 心跳断了，你怎么排查？

**面试官在考什么**：你有没有想过 gRPC 的生产可观测性问题。

**回答**：

> 这就是为什么我在 gRPC 服务器上注册了 **reflection service**：
>
> 1. **grpcurl**：`grpcurl -plaintext localhost:50051 list` 可以列出所有 RPC，`grpcurl -d '{"device_id":"xxx"}' localhost:50051 localpilot.v1.DeviceService/Heartbeat` 可以手动调用。这相当于 gRPC 的 curl。
> 2. **结构化日志**：Agent 和 Controller 都用 `slog`（Go 1.21+ 标准库），每条 gRPC 调用都有 key=value 的结构化日志。出问题时有 `device_id`、`error`、`consecutive_failures` 等字段可以直接 grep。
> 3. **心跳计数器**：Agent 的心跳循环记录了连续失败次数——3 次 WARN，6 次 ERROR+触发重连。出问题时不用连 gRPC，看 Agent 日志就知道网络状态。
> 4. **Dashboard 指标**：Controller 的 `/api/devices` 返回每台设备的 `last_heartbeat` 时间戳和 `state` 字段，出问题时 Dashboard 一眼就能看到哪台设备变红了。
>
> 实际上 gRPC 的"不好调试"是可以用工具链弥补的。强类型契约（proto）带来的编译期安全保障，远比调试时多看一步 grpcurl 的代价值得。

### Q3: gRPC 的 server-streaming 用来推送日志。但如果网络不稳定，stream 断了怎么办？已经推送过的日志会丢吗？Agent 怎么知道 Controller 有没有收到？

**面试官在考什么**：你对 gRPC streaming 的可靠性有没有深入思考。

**回答**：

> gRPC server-streaming 是单向的——Agent 推送，Controller 接收，没有应用层 ACK。确实存在三个问题：
>
> **"stream 断了怎么办？"** — Agent 的 `Execute` handler 检测 `stream.Context().Err()`。如果 stream 断了（Controller 断开或网络中断），context 会取消，executor 收到取消信号后终止命令进程。不会出现"Agent 一直往断了 stream 写数据"的情况。
>
> **"日志会丢吗？"** — 会。当前 Phase 2 的实现没有日志持久化——Agent 的日志只通过 gRPC stream 发送，Controller 收到后存内存。如果 stream 断了：
> - Controller 侧：已收到的日志在内存中，Dashboard 轮询可以看到。未收到的日志丢失。
> - 这是故意做的权衡——日志不是关键数据。任务的最终状态（COMPLETED/FAILED）和退出码是持久化到 SQLite 的。日志丢失不影响任务结果。
>
> **"Agent 怎么知道 Controller 收到了？"** — 当前不知道。这是有意为之——gRPC 底层是基于 TCP 的，TCP 保证了"发送方发出去了、接收方收到了"。应用层不重复确认。如果要做可靠日志传输，可以在 LogChunk 里加 `ack_required` 字段，Controller 收到后通过另一个 stream 或 unary 调用发 ACK——但这对视频转码场景来说是过度设计。
>
> 这是分布式系统中经典的"至多一次(at-most-once) vs 至少一次(at-least-once)"语义选择。日志选了 at-most-once——简单，够用。

## 二、调度器

### Q4: 你的调度器打分公式里，权重是写死的（CPU×0.35 + 内存×0.25 + ...）。谁告诉你这些权重是对的？如果实际运行中发现不对，你怎么调整？

**面试官在考什么**：你知道硬编码权重的局限，以及如何用数据驱动优化。

**回答**：

> 这些权重确实是经验的初始值，不是从数据中拟合出来的。如果实际运行发现不对，有三个调整层次：
>
> **调整 1：手动观察调参。** 在 Dashboard 上看调度器的选择是否合理——比如"明明设备 A 负载更低，为什么选了 B？"然后去 scorer.go 里调权重。这是当前 Phase 3 的做法。
>
> **调整 2：A/B 测试。** 同时跑两套权重，对比任务平均完成时间。比如权重组合 W1(0.35,0.25,0.20,0.10,0.10) vs W2(0.30,0.30,0.15,0.15,0.10)，看哪组产出的 COMPLETED 任务平均耗时更短。不需要改代码——把权重做成环境变量或配置文件。
>
> **调整 3：在线学习。** 记录每台设备执行每个任务的实际耗时，用梯度下降反向拟合最优权重。但这需要积累足够多的历史数据，对于一个"个人设备网格"的规模来说，在线学习可能是杀鸡用牛刀。
>
> 不过有个更根本的问题：**同一个权重对不同任务类型不一定最优**。视频转码是 CPU 密集型，应该提高 CPU 权重；大文件压缩是 IO 密集型，应该加磁盘权重；AI 推理需要 GPU，应该大幅提高 GPU 权重。这就是为什么 Phase 4 计划加入**任务级别的权重调整**——调度器先看任务类型，再选择对应的权重配置。

### Q5: 你说调度器"选第一个 ONLINE 设备"在 Phase 2 被替换成了多维打分。但如果只有一台设备呢？你的打分算法有意义吗？

**面试官在考什么**：你有没有意识到算法的退化场景。

**回答**：

> 只有一台设备时，打分结果永远是那台设备得分最高——算法在数学上正确但没有实际意义。但这不是 bug：
>
> 1. **调度器不增加开销**：打分只对候选设备列表计算，一台设备就是一次 O(1) 的乘法加法，耗时微秒级。没有性能损失。
> 2. **代码统一**：不管几台设备，调度路径都是 `worker → SelectDevice → Score → 返回最优`。不因为设备数量变化而走不同的代码分支——少了 if-else，少了 bug。
> 3. **如果真的只有一台设备，更值得关注的是另一个问题**：那台设备挂了就没地方迁移任务了。调度器的任务其实是**在设备不够的时候告诉用户"你的集群容量不足"**，目前 Phase 3 还没到这个层面。

### Q6: 你的调度器考虑"当前 CPU 使用率"，但 CPU 使用率是瞬时值。如果一台设备刚好在打分的那一刻 CPU 不高，但下一秒就飙满了怎么办？

**面试官在考什么**：你对调度中的"快照陈旧问题"有没有认识。

**回答**：

> 这是个真实问题——调度器基于心跳数据的快照做决策，心跳间隔是 5 秒。在两次心跳之间，设备负载可能剧烈变化。
>
> 有几个缓解措施：
>
> 1. **心跳携带的是瞬时值但更新频繁**：5 秒间隔在个人设备场景下已经足够——个人设备不会像云服务器那样在秒级有剧烈负载波动。
> 2. **可以引入滑动窗口平均值**：Controller 在内存中保留最近 3-5 次心跳的 CPU 数据，打分时用移动平均而不是最新值。这能平滑掉尖刺。代码里 `Device` 结构体已经预留了扩展字段的空间。
> 3. **可以在 Agent 侧做准入控制**：Agent 收到 Execute 请求时，再检查一次当前负载。如果负载突然飙高，可以拒绝执行（返回 resource_busy 错误），Controller 收到后重新调度到其他设备。
>
> 方案 3 是最好的，但因为涉及 Agent 和 Controller 之间的新增交互协议，留给 Phase 4 实现。当前方案在个人设备场景下的误判概率很低。

## 三、容错与任务迁移

### Q7: 设备离线 30 秒后你才标记 OFFLINE 并迁移任务。但如果一个任务只需要 5 秒就完成了，它可能已经在新设备上重新开始执行了——原设备上的任务还在跑，你怎么办？

**面试官在考什么**：你对"重复执行"和幂等性有没有思考。

**回答**：

> 这是分布式系统中经典的双主（split-brain）问题。当前 Phase 4 的处理是**尽力而为，不保证严格一次**（exactly-once）。具体来说：
>
> **场景**：设备 A 执行任务，心跳断连 → Controller 等待 30 秒 → 标记 OFFLINE → 迁移到设备 B。但设备 A 其实没死，只是网络断了——它的任务还在跑，甚至可能已经完成。
>
> **当前处理**：
> 1. 设备 A 重连后重新注册 → Controller 分配新的 device_id（或复用旧 UUID）→ 旧的 task 已经从 Controller 的 runningJobs 中移除（迁移到了 B）→ 设备 A 上的旧任务没有人收结果了——**被遗弃**。
> 2. 设备 B 从头执行任务——因为当前 Phase 4 没有 checkpoint，任务只能重新执行。
> 3. 用户可能看到同一个任务被执行了两次——这是 at-least-once 语义。
>
> **更好的方案**（留给后续）：
> - **幂等任务标记**：用户提交任务时声明"此任务可安全重复执行"（如视频转码——相同输入永远出相同输出）。调度器对幂等任务不额外处理，对非幂等任务加分布式锁。
> - **fencing token**：Controller 分配任务时附带一个递增的 epoch 号。Agent 在执行前和 Controller 确认"我的 epoch 是否仍然有效"。如果被迁移了，epoch 已经增加，Agent 的确认失败 → 放弃执行。
>
> 目前 Phase 4 没有解决这个问题，因为个人设备网格场景下，30 秒断连通常是设备真的挂了（断电、WiFi 断开），而不是短暂的网络分区。split-brain 的概率很低。

### Q8: 任务迁移后从零开始执行——之前的进度全丢了。为什么不做 checkpoint？

**面试官在考什么**：你对 checkpoint 机制的工程复杂度评估。

**回答**：

> Checkpoint 说起来简单（"定期保存中间状态"），做起来涉及三个层面的问题：
>
> **1. 谁来做 checkpoint？** 如果是 Agent 做，Agent 怎么知道任务的中间状态是什么？对于 `ffmpeg` 转码——只有在关键帧处才能切分，Agent 不理解 ffmpeg 的输出格式，没法做 checkpoint。除非**限制可执行的任务类型**（比如只允许特定的计算框架），但这会牺牲灵活性。
>
> **2. checkpoint 存哪里？** 存在 Agent 本地 → Agent 挂了 checkpoint 也丢了。存在 Controller → 每次 checkpoint 都是一次网络传输，大任务的 checkpoint 可能几百 MB。
>
> **3. 恢复时怎么验证 checkpoint 完整性？** 文件可能损坏、可能被篡改。需要校验和 + 版本号 + 重试机制。
>
> 所以 Phase 4 没有实现 checkpoint，因为这本质上是一个**分布式快照问题**——在 LocalPilot 的当前规模下，让任务从头执行比实现可靠的 checkpoint 更经济。大多数个人计算任务（视频转码、数据处理）是几分钟到几十分钟的——从头执行的代价比维护 checkpoint 基础设施的代价低。
>
> 如果面试官追问"那你怎么在面试中展示你的分布式系统功底"——
>
> 我会说："我清楚地知道 checkpoint 的工程复杂度，并且做了明确的 tradeoff 决策——在当前阶段不实现。这比'没想过这个问题'或者'盲目实现了有 bug 的 checkpoint'要好得多。"

## 四、沙箱安全

### Q9: 你的"沙箱"就是 os/exec + setrlimit——这不叫沙箱，这只是跑了个子进程。如果用户提交 `rm -rf /`，你的 Agent 会被删光吗？

**面试官在考什么**：你是否有安全意识，是否理解当前方案的局限。

**回答**：

> 会。当前 ProcessSandbox 和直接跑 `os/exec` 几乎没有安全隔离——子进程拥有和 Agent 完全相同的文件系统权限和网络权限。
>
> 这是故意的设计决策，不是疏忽：
>
> **LocalPilot 的威胁模型是"自己用自己的设备"**，不是"公共平台跑不可信代码"。用户在自己的旧 MacBook 上跑自己的 ffmpeg 转码——不存在恶意代码风险。在这个威胁模型下，基础资源限制（防止内存泄漏拖垮 Agent）已经覆盖了 90% 的实际问题。
>
> 如果需要真正的安全隔离：
> 1. **Docker 容器**：`Sandbox` 接口已经定义好了，实现 `DockerSandbox` 就可以——只读根文件系统 + 网络隔离 + tmpfs 临时写入。Phase 2 后期。
> 2. **Linux cgroups + namespaces**：不需要 Docker daemon，比容器更轻量。但需要 root 权限和 Linux。
> 3. **WASM 微隔离**：WebAssembly 运行时（如 Wasmtime）提供比容器更细粒度的沙箱。但只支持编译到 WASM 的语言（Rust/C/Go），不支持 ffmpeg 等原生二进制。
>
> 如果一个面试官拿这个攻击你——你的回应应该是"我的威胁模型决定了我的安全投入。如果你的威胁模型不同，我的架构预留了 DockerSandbox 的扩展点。"

## 五、架构设计

### Q10: 你的系统有三个组件——Agent、Controller、Dashboard。为什么不做成两个？Agent 直接内置一个 Web 界面，去掉 Controller，不是更简单？

**面试官在考什么**：你理解"中心化调度 vs P2P"的 tradeoff。

**回答**：

> 去掉 Controller 意味着从**中心化调度**变成**去中心化 P2P**。这在工程上会引入三个新问题：
>
> **1. 全局状态一致性**：没有 Controller，每台 Agent 需要自己维护集群中所有设备的列表。设备上下线需要通过 P2P gossip 协议同步——复杂度远高于一个中心化注册表。
>
> **2. 调度决策冲突**：两台设备同时看到同一个任务，都自认为"我最适合执行"，然后都开始执行——结果重复。需要一个分布式共识算法（Raft/Paxos）来解决"谁执行这个任务"。
>
> **3. Dashboard 视角**：如果每台 Agent 都有自己的 Web 界面，用户要在三个浏览器 Tab 之间切换来管理三台设备。用一个中心化的 Dashboard 连一个 Controller，用户体验好得多。
>
> 中心化的"缺点"是 Controller 是单点故障——但如果 Controller 挂了，说明主力机挂了，整个集群本来也就不工作了（因为 Agent 们的计算任务需要 Controller 下发）。在这个场景下，Controller 的单点故障不是问题——主力机是用户正在用的设备，挂了用户自己能感知。
>
> 所以中心化架构在 LocalPilot 的场景下是合理的设计——KISS 原则（Keep It Simple, Stupid）。

### Q11: 你说"Proto 是系统的单一事实来源"，但你的 Dashboard TypeScript 类型是手写的（`types/index.ts`），不是从 proto 生成的。如果后端改了一个字段名，前端怎么知道？

**面试官在考什么**：你对"契约一致性"的实践。

**回答**：

> 这是一个真实的技术债。当前确实是手写同步——改了 proto 里的 `DeviceInfo` 字段，需要手动去 `types/index.ts` 和 `device.go` 的 json tag 两处同时更新。漏了任何一处就会出运行时 bug。
>
> 在生产项目中应该用工具链解决：
> - **protoc-gen-ts**：从 proto 自动生成 TypeScript 类型，和 Go 的 `protoc-gen-go` 一样。类型不一致会在编译期报错，而不是运行时。
> - **CI 检查**：在 CI pipeline 里加一步——对比 proto 生成的 TS 类型和手写的 `types/index.ts`，如果不一致就阻止合并。
>
> 当前没做是因为 Phase 0-3 在快速迭代，proto 字段还在频繁变动。手写类型虽然容易出错，但在一个人开发的场景下，迭代速度比正确性更重要。Phase 5 应该引入 protoc-gen-ts。
>
> 这其实是"**技术债意识**"的问题——你不需要一开始就完美，但你需要知道你的不完美在哪里，以及什么时候该还债。
